# VibeSeek 实测评测报告

> 真实 DeepSeek API 跑出,非模拟。所有用例**可执行验证**(把模型产出的代码 import 进来真跑),且每套用例都有"参考解必过、原始必挂"的机制自检,排除评测器造假。原始数据归档于 `packages/core/evals-*.json`。
>
> 跑测日期:2026-06-13 · 总花费约 ¥54。

## 一、能力:32 个编程任务 × 三种模式

32 例覆盖改 bug / 加功能 / 写测试 / 重构四类,含 8 个边界密集的难任务(罗马数字、CSV RFC-4180 转义、多文件管道改造、状态机、深路径取值、宽松邮箱校验等)。每例在 `auto` / 纯 flash / 纯 pro 三种模式下各跑 2 轮(共 192 次):

| 模式 | 通过率 | 相对成本 |
|---|---|---|
| 自动路由 | **100%** (64/64) | 比纯 pro 省 **60%** |
| 纯 v4-flash | **100%** (64/64) | 最低 |
| 纯 v4-pro | **100%** (64/64) | 基准 |

**结论**:自动路由(简单任务走 flash、复杂升 pro)以纯 pro **40% 的成本**拿到了同等的 100% 通过率。flash 单独也做对了全部难任务——DeepSeek 在定义清晰的编程任务上很能打。
*(说明:100% 也意味着这套题尚未触及能力上限;要找到极限需要真实大型仓库与模糊需求的任务,属于后续工作。)*

## 二、全库模式:大项目里"只说要什么、不说在哪"

合成的 ~220K token 大项目,植入 3 个只描述行为、不点名文件的任务(跨文件找并修 bug、改配置常量、给既有模块加函数),全库模式 vs 传统检索,pro,共 72 次:

| | 通过率 | 检索调用 | 缓存命中 |
|---|---|---|---|
| 全库模式 | **72/72 (100%)** | **0 次** | 99.96% |
| 传统检索 | 100% | 平均 2-3 次/任务 | 82% |

**结论**:全库模式把整个项目放进上下文,模型无需反复 grep/读取就直接定位并改对,检索调用归零。

## 三、缓存命中率(随上下文规模上升)

命中率 = 已缓存前缀 ÷ 总输入,所以它取决于场景,不是一个固定数:

| 场景 | 命中率 |
|---|---|
| 小型一次性任务(单文件、3-5 轮) | 88-90% |
| 持续会话(同一项目连续编辑,逐轮爬升) | 单轮升至 **95%**,累计 91% |
| 全库 / 大上下文 | **99.96%** |

**结论**:上下文越大、会话越长,稳定前缀占比越高,命中率越高。日常编码会话稳定在 90-95%,大型项目场景 99%+。我们如实给出分场景数字,而不是一个扁平的高百分比——小任务那部分未命中是每次真正新增的内容,属于数学下限,无法也不必优化。
